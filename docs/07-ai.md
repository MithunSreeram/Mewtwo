# Mewtwo — AI Module

The AI module lives in `mewtwo/modules/ai/`. It's a thin wrapper around the
Anthropic SDK that handles workspace context injection, forced tool-use for
structured output, and all the prompt templates for each task.

---

## Design: Forced Tool-Use Instead of JSON Parsing

Every structured AI call in Mewtwo uses `tool_choice={"type": "tool", "name": "..."}`:

```python
def complete_with_tool(self, system, messages, tool, max_tokens=4096) -> dict:
    response = self.client.messages.create(
        model=self.model,
        max_tokens=max_tokens,
        system=system,
        messages=messages,
        tools=[tool],
        tool_choice={"type": "tool", "name": tool["name"]},  # force this tool
    )
    for block in response.content:
        if block.type == "tool_use":
            return block.input   # already a validated dict
    return {}
```

`tool_choice={"type":"tool","name":"..."}` forces Claude to respond by calling
that specific tool — it cannot output prose. The `block.input` is already the
parsed, schema-validated dict. No JSON parsing, no regex, no error handling for
malformed output.

Contrast with the naive approach:
```python
# This breaks any time Claude says "Here is the JSON:" before the block
result = json.loads(response.content[0].text)
```

## client.py — AIClient

### Initialisation

```python
class AIClient:
    def __init__(self, api_key=None, model=None):
        key = api_key or config.anthropic_api_key()
        if not key:
            raise RuntimeError("ANTHROPIC_API_KEY not set.")
        self.client = anthropic.Anthropic(api_key=key)
        self.model = model or config.model()
```

`config.anthropic_api_key()` reads from the environment or a `.env` file in the
workspace. `config.model()` defaults to `claude-3-5-sonnet-20241022` but can be
overridden with `MEWTWO_MODEL`. This lets you switch to Haiku for faster/cheaper
runs without changing code.

### Three completion modes

**1. `complete()` — non-streaming, returns text**
```python
response = self.client.messages.create(model=..., system=..., messages=...)
for block in response.content:
    if block.type == "text":   return block.text
    if block.type == "tool_use": return json.dumps(block.input)
```
Handles both text and tool_use response types. Used for free-form tasks like
`write_executive_summary`.

**2. `stream()` — yields text deltas**
```python
with self.client.messages.stream(...) as stream:
    yield from stream.text_stream
```
Used by `mewtwo ai ask` to print Claude's response incrementally. `stream.text_stream`
is a generator of string chunks from the Anthropic SDK.

**3. `complete_with_tool()` — forced structured output**
```python
tool_choice={"type": "tool", "name": tool["name"]}
→ block.input  # validated dict matching tool's input_schema
```
Used for all structured tasks: surface expansion, triage, enrichment, payloads, chains.

### Workspace context injection

```python
def workspace_context_snippet(self, db_path=None) -> str:
    db = get_db(db_path)
    targets = list(db["targets"].rows)
    target = targets[0] if targets else {}
    summary = {
        "target": target.get("name"),
        "platform": target.get("platform"),
        "subdomains": db["subdomains"].count,
        "urls": db["urls"].count,
        "attack_vectors": db["attack_vectors"].count,
        "findings": db["findings"].count,
    }
    return json.dumps(summary, indent=2)
```

This snippet is injected into every system prompt. It gives Claude context about
the current engagement — number of URLs, findings, platform — so AI suggestions
are calibrated to the target's size and scope.

### Task methods

Each method composes system prompt + user message + tool, then calls
`complete_with_tool`:

```python
def enrich_finding(self, finding_dict, db_path=None) -> dict:
    system = enrichment_system(self.workspace_context_snippet(db_path))
    user_msg = enrichment_user(finding_dict)
    return self.complete_with_tool(system, [{"role":"user","content":user_msg}],
                                   ENRICHMENT_TOOL, max_tokens=8096)
```

`max_tokens=8096` for enrichment because reproduction steps and remediation can be
long. Other tasks use 4096.

## tools.py — Tool Schemas

Each tool schema defines what Claude must return. The schema is JSON Schema format.

### ATTACK_VECTORS_TOOL

```python
{
    "name": "report_attack_vectors",
    "input_schema": {
        "type": "object",
        "properties": {
            "vectors": {
                "type": "array",
                "items": {
                    "properties": {
                        "category": {"type": "string", "enum": [
                            "authentication", "authorization", "injection",
                            "ssrf", "info_disclosure", "business_logic",
                            "client_side", "configuration"
                        ]},
                        "title": {"type": "string"},
                        "url": {"type": "string"},
                        "risk_rating": {"type": "string", "enum": ["critical","high","medium","low"]},
                        "rationale": {"type": "string"},
                    },
                    "required": ["category","title","description","url","risk_rating","rationale"]
                }
            }
        }
    }
}
```

The `enum` on `category` prevents Claude from inventing new categories. The 8 values
match `VectorCategory` exactly — so the returned strings are safe to pass directly
to `VectorCategory(value)`.

### TRIAGE_TOOL

```python
{
    "name": "triage_finding",
    "properties": {
        "is_finding": {"type": "boolean"},
        "severity": {"enum": ["critical","high","medium","low","informational"]},
        "confidence": {"enum": ["confirmed","likely","possible","unlikely"]},
        "reason": {"type": "string"},
        "follow_up": {"type": "string"},
    }
}
```

Two-level classification: `is_finding` (yes/no) and `confidence` (how sure). A
`possible` IDOR with `is_finding: true` is treated very differently from a
`confirmed` XSS.

### ATTACK_CHAIN_TOOL

```python
{
    "name": "report_attack_chains",
    "properties": {
        "chains": {
            "items": {
                "properties": {
                    "title": ...,
                    "combined_severity": {"enum": ["critical","high","medium","low"]},
                    "finding_ids": {"type": "array", "items": {"type": "string"}},
                    "attack_narrative": {"type": "string"},
                    "business_impact": {"type": "string"},
                    "prerequisites": {"type": "string"},
                }
            }
        }
    }
}
```

`finding_ids` links chain members back to specific DB rows. A chain is only useful
if you can trace it back to confirmed findings.

### PERSONALISED_PAYLOAD_TOOL

```python
{
    "items": {
        "properties": {
            "payload":   {"type": "string"},
            "technique": {"type": "string"},
            "rationale": {"type": "string"},  # why this payload fits the stack
            "placement": {"type": "string"},  # query param / header / body
        }
    }
}
```

Unlike `generate_payloads` (which returns plain strings), this returns structured
objects. The `rationale` explains why a specific payload is relevant to the detected
tech stack — so you understand what you're sending, not just paste it blindly.

## prompts.py — Prompt Templates

All prompts are functions, not module-level strings. This keeps them testable and
makes it easy to pass context into them.

### Shared persona

```python
_PURPLE_TEAM_PERSONA = """You are a senior security researcher with deep offensive
and defensive expertise. You think like a red teamer discovering vulnerabilities
but document like a blue teamer producing clear, actionable security reports.
You are thorough, precise, and avoid false positives."""
```

This persona is prepended to every system prompt. "Avoid false positives" is
important — it pushes Claude toward conservative triage rather than over-reporting.

### Context injection pattern

```python
def enrichment_system(workspace_context: str = "") -> str:
    ctx = f"\n\nWorkspace context:\n{workspace_context}" if workspace_context else ""
    return f"""{_PURPLE_TEAM_PERSONA}
    You write professional bug bounty vulnerability reports.{ctx}
    ..."""
```

The workspace context is optional — if the DB isn't available, the prompt still
works, just without the target metadata.

### Enrichment prompt structure

```python
def enrichment_user(finding: dict) -> str:
    return f"""Enrich this vulnerability finding for a bug bounty submission:

Title: {finding.get('title')}
Vulnerability class: {finding.get('vuln_class')}
Severity: {finding.get('severity')}
Affected URL: {finding.get('url')}
Affected parameter: {finding.get('parameter', 'N/A')}

Raw evidence / notes:
{finding.get('description') or finding.get('evidence') or 'No evidence provided yet'}

Use the provided tool to return enriched description, impact, reproduction steps, and remediation."""
```

Structured fields first, then raw evidence — Claude can improve/expand what's
there rather than inventing from scratch.

### Attack chain prompt

```python
def attack_chain_system(workspace_context="") -> str:
    return f"""...
Classic chains to look for:
- CORS misconfig → stored XSS → session hijack
- SSRF → cloud metadata → credential theft → privilege escalation
- IDOR + weak auth → account takeover
- Open redirect → OAuth token theft
- Info disclosure (stack trace/debug) → targeted SQLi or XXE
- Rate limit bypass + user enumeration → credential stuffing

Report chains that are realistic and exploitable, not theoretical."""
```

Seeding the system prompt with known chain patterns acts as a few-shot example —
Claude recognises these structures in the actual findings list.

### Exec summary

```python
def exec_summary_user(target_name, findings) -> str:
    severity_counts = {}
    for f in findings:
        sev = f.get("severity", "informational")
        severity_counts[sev] = severity_counts.get(sev, 0) + 1

    top_findings = sorted(findings, key=lambda f: {
        "critical": 0, "high": 1, "medium": 2, ...
    }.get(f.get("severity"), 5))[:5]
```

The summary prompt pre-processes the findings list server-side — passes severity
counts and only the top 5 findings to keep the prompt small. An 80-finding list
would push the prompt over max tokens and also confuse the narrative.

### `mewtwo ai chains`

```python

    """Identify attack chains linking multiple findings into higher-impact scenarios."""
    findings = repo.for_target(target_row["id"])

    for chain in result.get("chains", []):

```

Only `confirmed` and `draft` findings are passed — informational notes and
false positives would produce noise chains.
